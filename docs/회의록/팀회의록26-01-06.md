## 주요 안건 및 논의사항

### 1. 팀 목표 설정
**핵심 목표**:
- 고급 과제를 포트폴리오로 적극 활용
- 과제 완료 후 포트폴리오 수준의 결과물 도출
- 1단계 구현 목표: 텍스트만을 입력하여 광고 컨텐츠(이미지, 문구 등) 생성
- 2단계 구현 목표: 추가 기능을 통한 품질 개선

### 2. 핵심 논의사항
**주요 이슈**:
- (백엔드) 플로우 시퀀스 다이어그램에 대한 논의
    - 백엔드에서 각 단계에 호출하고 콜백받는 것이 비효율적이라는 의문 제기
    - services.py에서 백엔드 중심의 오케스트레이션 패턴이 가지는 장점
    1. 중앙 집중식 제어 관리
        - 백엔드 API가 모든 단계를 조율하여 전체 플로우를 한 곳에서 관리 가능
        - 프로세스 변경이나 새로운 단계 추가 시 백엔드만 수정하게 되어 유지보수 용이
    2. 보안 강화
        - 프론트엔드는 백엔드API만 호출하여 백엔드에서 관리하는 외부 API키와 민감한 정보들은 클라이언트에 노출되지 않습니다.
    3. 에러 핸들링 및 복구 용이성
        - 각 단계별 실패 시 백엔드에서 재시도, 롤백, 대체 로직 등을 중앙에서 처리할 수 있습니다. 클라이언트는 복잡한 에러 처리 로직 없이 간단한 결과만 받으면 됩니다.
    4. 비용 및 성능 최적화
        - 백엔드에서 각 API 호출을 최적화하고, 캐싱을 적용하거나, 불필요한 중복 호출을 방지할 수 있습니다. 클라이언트별로 개별 호출하는 것보다 효율적입니다.
    5. 모니터링 및 로깅
        - 모든 트랜잭션이 백엔드를 거치므로 전체 프로세스를 한 곳에서 모니터링하고 로그를 수집할 수 있습니다. 문제 발생 시 추적과 디버깅이 훨씬 쉽습니다.
    6. 확장성 및 유연성
        - 새로운 AI 모델이나 서비스를 추가할 때 프론트엔드 변경 없이 백엔드에서만 통합하면 됩니다. 여러 클라이언트(웹, 모바일)가 동일한 백엔드 API를 공유할 수 있습니다.
    7. 프론트엔드 단순화
        - 프론트엔드는 복잡한 비즈니스 로직이나 여러 API 간의 데이터 변환을 처리할 필요 없이 단순히 사용자의 입력을 받고 결과를 표시하는 역할만 수행하면 되기 때문에 코드가 간결해집니다.
    8. 비즈니스 로직 보호
        - 광고 생성 프로세스의 핵심 로직이 백엔드에 있어 지적 재산권을 보호할 수 있습니다. 클라이언트는 디컴파일이 가능하지만 서버 코드는 보호됩니다.
    - 이러한 장점들로 인해 복잡한 AI 워크플로우나 여러 외부 서비스를 통합하는 시스템에서는 백엔드 중심 오케스트레이션이 일반적으로 선호되는 아키텍처 패턴입니다.
