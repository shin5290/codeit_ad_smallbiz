솔직히 나는 창의적인 생각을 주로하는 타입이 아니며 데이터 적으로 어떤게 좋은 도메인이고 어떤게 좋은 아이디어인지 파악할 수 없음

하지만 지금 문서들을 보면서 논의할만한 주제들을 몇개 찾아낼 수 있었고 그걸 기준으로 이야기를 해볼까 함

우선적으로 승목님은 어떤 아이디어의 제안이라기 보다는 프롬프트 전략의 제안이라 배현석님 문서를 위주로 확인하였음

---

## 첫번째, 문제의 방향성

- 전반적인 개발 진행이 느린 것은 도메인의 문제가 아님
- 좋은 서비스에 대한 고민은 프로젝트 내내 해야할 문제 상황인식과 적절한 판단이지만, 기초적인 기능 구현이 안되어있으면 당연히 좋은 서비스를 고민할 수 없음
- 현재의 이 고민과 회의는 당연히 매우 필요하고 중요한 고민이지만, "사용자의 요구에 입각하여 프롬프트를 생성하고 그에 맞는 적합한 이미지를 생성한다" 라는 기초적인 과정없이 너무 결과를 보고 있는 느낌을 지울 수 없음

## 두번째, 도메인의 문제

현재 나와있는 대부분의 이미지 모델은 서양, 중국쪽에 치우쳐있음, 다시 말하면 한국에 특화된 광고 도메인을 선정하기에는 솔직히 한계가 있긴 함
그렇다고 T2I를 보조로두고 I2I로 가기에는 몇가지 문제가 있는데
1. 사용자가 찍은 이미지가 화질이 너무 안좋은경우
   - 이 경우에는 어떠한 방법으로든 업스케일을 진행해야하는데, 전처리로 업스케일을 진행한다고 해서 제대로 사용자가 원하는 퀄리티의 이미지가 나올 수 있을 지 의문
2. 사용자가 찍은 이미지의 배경이 너무 어지러운 경우
	- 사용자가 찍은 이미지를 기반으로 새로운 배경을 만들어 붙이기 위해선 rembg같은 라이브러리를 이용해서 배경을 제거하고 이미지를 합성하는 작업이 필요한데, rembg는 배경제거 퀄리티가 나쁘지 않은것으로 알고는 있지만, 배경이 조금만 어지러워도 퀄리티가 어떻게 될지 장담할 수 없음
3. 사용자가 T2I를 원할 가능성
	- T2I를 보조로 둔다고 가정했을 때 어쨌든 원하는 사용자가 있을 것이고 그렇다는건 T2I로도 적어도 한국에 맞는 이미지정도는 뽑아내야함 그러려면 LoRA학습이 진행되어야하고 LoRA를 학습하기 위해 필요한 데이터셋 모으기와 학습 후에 테스트 등의 문제를 고려하면 T2I는 기능적으로만 보조이지 구현적으로 주객전도 될 가능성이 높음
4. 절망적인 텍스트 퀄리티
	- 우선 이미지 모델이 뽑아주는 텍스트의 퀄리티는 절망적임, 영어는 그나마 나을 수 있으나 한국어는 매우 절망적임, 그렇기에 필요한 것은 텍스트 오버레이 기능을 활용해야함, 하지만 텍스트 오버레이 기능을 활용하는 것도 한계가 있는데, 텍스트 오버레이를 이용하면 디자인적인 요소는 포기해야함, 그로인해 발생하는 퀄리티의 저하에 대해 대책을 마련해야할 것

---

## 결론

- 아이디어는 아니지만 기술적으로 기능적으로 어떤 부분에서 한계가 있는지에 대해서도 명확히 할 필요가 있다고 생각
- 당연히 참신하고 좋은 아이디어는 좋은 프로젝트를 만드는 데에 중요하고 의견 하나하나 소중하다는 것은 알지만, 현재 가지고 있는 기술력과 자원, 한계를 명확히하는게 더 우선일 것이라고 생각함

---

## 제안

1. 도메인 확정 전에, 먼저 "사용자 입력 → 프롬프트 생성 → 이미지 생성" 기본 파이프라인을 한 번이라도 완성 및 테스트를 통해 한계 설정
2. 그 과정에서 T2I/I2I 각각의 실제 퀄리티를 테스트해보고, 어느 쪽에 집중할지 데이터 기반으로 결정
3. 특히 rembg 배경 제거, 텍스트 오버레이 퀄리티는 실제 샘플 이미지로 검증이 필요
